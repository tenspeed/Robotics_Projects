115  * as an input; MOSI, SCK, and SS are outputs. Note that even though SS is often not
116  * used by this routine, it needs to be set as an output, otherwise if anything drives
117  * it low, the SPI port automatically reconfigures itself as a slave and stops working
118  * as an interface to the SD card. Things seem more reliable with the pullup on MISO
119  * turned on also.

170 /** This class implements an interface to an SD card. Data is stored up in a buffer,
171  * then written to the card when the buffer becomes full or when the send_now
172  * modifier is inserted into the SD card object's output stream. (Alternatively, one
173  * can specify that carriage returns cause writing to the card using the macro
174  * SD_SEND_NOW_CHAR in sd_card.h.) The SPI interface to the SD card is used, as it is
175  * much simpler and requires less wires than the full high-speed interface. The SPI
176  * interface is a bit slow, though.
177  *
178  * The ELM-FAT-FS library is used for its low-level and FAT file system drivers. This
179  * library can be found at http://elm-chan.org/fsw/ff/00index_e.html. The filesystem
180  * library interfaces with the sd_card class through two sets of C functions. The
181  * first set of functions is in files ff.h and ff.c; these functions provide the
182  * functionality of a file system: initializing and mounting volumes, opening files
183  * and directories, reading and writing data, and so on. The second set of functions
184  * is a set of callback functions which are provided in sd_card.cc. Written in C,
185  * these callbacks are called by the ELM-FAT-FS library to provide platform and media
186  * specific access; they allow the AVR processor to work with the SD card. Most of
187  * the C functions in sd_card.cc are slightly modified versions of functions from the
188  * AVR-specific example files distributed with the ELM-FAT-FS library.
189  *
190  * \section sd_prep_card Preparing an SD Card for Use
191  * The SD cards used by this program should be formatted as MS-DOS style volumes in
192  * the "floppy disk" style, not the "hard disk" style. This means that there should
193  * not be a partition table, with one or more partitions each having a filesystem.
194  * Instead, the disk's main volume should be formatted. Many SD cards come formatted
195  * in the wrong style when purchased; the following steps will help determine if a
196  * card is formatted in the right or wrong way:
197  * \li Plug the card into a Linux computer.
198  * \li Open the card in a file browser; ignore the contents and close the browser.
199  * Opening the card caused it to be automatically mounted, which is why it was
200  * worth doing.
201  * \li Open a terminal window (konsole or gnome-terminal) and type 'mount'. Look
202  * near the end of the bunch of text which comes up for a line that looks like
203  * this: \n <tt>/dev/sde1 on /media/disk type vfat
204  * (rw,nosuid,nodev,uhelper=hal,uid=1000,utf8,shortname=mixed,flush)</tt>
205  * \li If the code <tt>/dev/xxxx</tt> on the left of the line ends with a number,
206  * such as <tt>/dev/sde1</tt>, you have the wrong kind of formatting; it will
207  * need to be reformatted. If the code has no number, such as <tt>/dev/sdf</tt>,
208  * then you should be able to use the card as is.
209  * \li The device name is usually <tt>/dev/sdxx</tt>, but on some computers the name
210  * may be different, such as <tt>/dev/mmcblk0</tt>. In this case, partitions may
211  * be named like <tt>/dev/mmcblk0p1</tt>, and the device name is actually
212  * <tt>/dev/mmcblk0</tt> including the "0" at the end.
213  * \li Unmount the card. In KDE, use the device mounter's "eject" button; in Gnome
214  * right-click on the device icon and choose unmount.
215  *
216  * The following process will format an SD card in "floppy disk style" on Linux:
217  * \li Get to a Linux computer on which you have root (sudo) access. You need root
218  * access in order to write directly to the SD card.
219  * \li Plug the card into an SD/MMC slot in the computer. Do not open it up in a
220  * file manager or mount it.
221  * \li Repeat the steps for checking the card's formatting if necessary to find the
222  * device name <tt>/dev/xxxn</tt> for the card. Note that in <tt>/dev/xxxn</tt>,
223  * the "\c xxx" is letters (for example sdf) and the "\t n" is a number, almost
224  * always \c 1.
225  * \li <b>WARNING:</b> You need to be very careful in the following command: <b>Do
226  * not enter the designation of one of your computer's hard drives</b>, as this
227  * command would then reformat your system drive and effectively wipe out all
228  * the data on that drive.
229  * \li Run the following command: \n
230  * <tt>sudo mkfs -t msdos -I /dev/xxx</tt> \n Note that
231  * the number is left off the device name. What you are doing is formatting the
232  * disk without any partitions; the numbers referred to partitions on the disk.
233  * \li Remove the card and re-insert it in the computer. Make sure you can open it
234  * in a file manager as usual.
235  * \li Unmount the card, remove it from the PC, and you're ready to use it to save
236  * some data.
237  *
238  * \section sd_card_drv Using the SD Card Driver Software
239  * In order to use the SD card driver, one must have an AVR board with an SD card
240  * slot, and the board must be configured to operate with Vcc at 3.3 volts. It is
241  * possible that multiple SD cards could be used on the same board, but this has
242  * not been tested, and it would be rather weird to want to do so. Hot swapping the
243  * SD card is possible, but it's usually not a very good idea because of hardware
244  * considerations: SD cards have big internal power supply capacitance and tend to
245  * inflict brownouts on the AVR when plugged in. Also, making sure that all your
246  * reads and writes are finished before swapping the card can be a problem.
247  *
248  * Configuration is done by modifying values in sd_card.h and ffconf.h. Recommended
249  * settings for ffconf.h (which configures the filesystem drivers) include the
250  * following:
251  * <table border="0" cellspacing="1">
252  * <tr><td> _FS_TINY </td><td> 0 </td><td> unless you have >= 4K SRAM </td></tr>
253  * <tr><td> _FS_READONLY </td><td> 0 </td><td> to enable writing </td></tr>
254  * <tr><td> _FS_MINIMIZE </td><td> 1 </td><td> to save space </td></tr>
255  * <tr><td> _USE_STRFUNC </td><td> 0 </td><td> unless you need strings </td></tr>
256  * <tr><td> _USE_MKFS </td><td> 0 </td><td> unless you need to format </td></tr>
257  * <tr><td> _USE_FORWARD </td><td> 0 </td><td> unless you need...this </td></tr>
258  * <tr><td> _CODE_PAGE </td><td> 1 </td><td> for ASCII coding </td></tr>
259  * <tr><td> _USE_LFN </td><td> 0 </td><td> to use short (8.3) filenames </td></tr>
260  * </table>
261  *
262  * Recommended settings for sd_card.h include the following:
263  * <table border="0" cellspacing="1">
264  * <tr><td> SD_BLOCK_SIZE </td><td> 512 </td>
265  * <td> This must be 512 for all SD cards </td></tr>
266  * <tr><td> SD_LINE_BUF_SIZE </td><td> 256 </td>
267  * <td> 256 Anything from about 16 to 512 should work; this value
268  * can be adjusted to balance speed with memory usage. </td></tr>
269  * </table>
270  *
271  * The processs of reading or writing a file consists of the following steps:
272  * \li Mount the card. This is Unix-talk for getting data from the boot record
273  * about where the partitions and base directories on the card can be found.
274  * \li Open a file using a method such as open_file_readonly(),
275  * open_file_overwrite(), etc.
276  * \li If lots of data is being logged, one can use the open_new_data_file() method
277  * which automatically creates file names in a sequence.
278  * \li Use the overloaded shift operator "<<" to write data to the file, or use the
279  * more primitive methods getchar(), putchar(), and puts() to read and write
280  * characters from and to the file.
281  * \li If at all possible, use the close_file() method to make sure all the data is
282  * saved to the file. If a file is not closed properly, some data may be left
283  * in the memory buffer and not written to the disk.
284  *
285  * \section sd_tested Functions tested and working
286  * \li Opening an SD card formatted without a partition
287  * \li Automatically creating data files in the root directory and writing to them
288  *
289  * \section sd_bugs Bugs
290  * \li When using SD cards formatted with one or more partitions, sometimes the
291  * files written by this driver aren't visible to a computer reading the card.
292  * Workaround: Make sure to use cards which don't have partitions on them.
293  * \li Writing to the SD card can be very slow, and this code does not attempt to
294  * prevent blocking of the rest of the system during card writes.
295  * \li Long file names have not been tested. They're probably not a good idea for
296  * a small embedded system anyway.
297  *
298  * \section sd_int_comp Internal stuff
299  * The DSTATUS flag byte, given in sd_card.c, holds some status bits for the disk:
300  * \li 0x01: STA_NOINIT
301  * \li 0x02: STA_NODISK
302  * \li 0x04: STA_PROTECT
303  *
304  * The FRESULT enumeration, given in ff.c, holds the result of a file operation.
305  * Its values are:
306  * \li 0: FR_OK
307  * \li 1: FR_DISK_ERR
308  * \li 2: FR_INT_ERR
309  * \li 3: FR_NOT_READY
310  * \li 4: FR_NO_FILE
311  * \li 5: FR_NO_PATH
312  * \li 6: FR_INVALID_NAME
313  * \li 7: FR_DENIED
314  * \li 8: FR_EXIST
315  * \li 9: FR_INVALID_OBJECT
316  * \li 10: FR_WRITE_PROTECTED
317  * \li 11: FR_INVALID_DRIVE
318  * \li 12: FR_NOT_ENABLED
319  * \li 13: FR_NO_FILESYSTEM
320  * \li 14: FR_MKFS_ABORTED
321  * \li 15: FR_TIMEOUT
322  *
323  */
