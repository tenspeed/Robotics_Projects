37 /** This enumeration lists the possible operational states of a task. These states are
38  * not the same as the states which are programmed by the user; these states are only
39  * used by the task scheduler to figure out which task to run at a given time
40  * according to priorities (if used) and which tasks are ready to be run.
41  */
42 
43 enum task_op_state
44 {
45  TASK_RUNNING, ///< The task's run() function is executing
46  TASK_PENDING, ///< The task needs to run again as soon as possible
47  TASK_WAITING, ///< The task is waiting for its next run time to occur
48  TASK_BLOCKED, ///< The task cannot run because a resource is unavailable
49  TASK_SUSPENDED ///< The task is turned off until its resume() is called
50 };

54 /** This class implements the behavior of a task in the context of a multitasking
55  * system. Each task runs "simultaneously" with other tasks. This means, of course,
56  * that execution switches quickly from one task to another quite quickly under the
57  * control of the main execution scheduler. Since a task can be in any of several
58  * states at a given time, a task object will have a list of state methods which
59  * can be run and be able to switch between those states.
60  *
61  * \section task_usage Usage
62  * The programmer uses this class by deriving a descendent class from it. Then
63  * s/he creates a run() method in the new class which implements the states.
64  * In the run() method, there's a big switch statement which chooses which state
65  * code is to run when the run() method is called. Checking the time and deciding
66  * when to run is taken care of by the schedule() method, which only runs the
67  * task's state code when the task's time interval has been reached; if the time
68  * interval is set to 0, the state code is run every time run() is called. State
69  * transitions are detected and tracked by this class for testing and debugging
70  * purposes.
71  *
72  * \section task_options Options
73  * \li Serial port debugging can be enabled by defining SERIAL_DEBUG and calling
74  * the constructor which is given a pointer to a uart object. The task will
75  * then write information about state transitions to the serial port during
76  * operation. This option is intended to be used during debugging, then turned
77  * off for production code, as serial device writing takes up time and memory,
78  * and of course it requires a serial device to be present and connected.
79  * \li Execution time profiling can be enabled by defining STL_PROFILE. This
80  * option causes the execution times of the state functions to be measured
81  * and a simple set of performance data to be kept. Performance data can be
82  * written to a serial port at a convenient time, generally after the system
83  * has been run in test for a while.
84  *
85  * \section task_intrn Internal Organization
86  * At any time, a task is in both a <i>user state</i> and an <i>operational
87  * state</i>. The user state is the state which the user manipulates; it chooses
88  * the action that the run() method will take each time it is called by the
89  * scheduler. The user state is the state which appears in the user's state
90  * transition diagrams. The operational state shows how the scheduler is treating a
91  * task - whether the task is running, waiting for its time to run, ready to run,
92  * suspended, etc. Operational states for all tasks follow the "state diagram"
93  * below.
94  *
95  * \dot
96  * digraph op_states {
97  * node [ shape = ellipse, fontname = Arial, fontsize = 10];
98  * waiting [ label = "Waiting " ];
99  * suspended [ label = "Suspended "];
100  * pending [ label = "Pending " ];
101  * running [ label = "Running " ];
102  * start [ label = "", shape = "plaintext" ];
103  * edge [ arrowhead = "normal", style = "solid", fontname = Courier, fontsize = 10 ];
104  * start->waiting [ label = " start " ];
105  * waiting->running [ label = " run() " ];
106  * running->waiting [ label = " run() finished " ];
107  * waiting->suspended [ label = "suspend() " ];
108  * suspended->waiting [ label = "resume() " ];
109  * running->pending [ label = "run_again_ASAP() " ];
110  * pending->running [ label = " run() " ];
111  * }
112  * \enddot
113  * Blocked tasks (due to resource contention) are not currently implemented.
114  *
115  * When a task is run cooperatively (rather than by interrupts), it is run when the
116  * method schedule() is called from within the main while loop in the main()
117  * routine. The schedule() method belongs to class stl_task and generally does not
118  * need to be modified by the user.
119  */
